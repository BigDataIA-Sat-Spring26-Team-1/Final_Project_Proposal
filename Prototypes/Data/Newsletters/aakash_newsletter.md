### 4. Hackers slipped a trojan into the code library behind most of the internet. Your team is probably affected
**Source:** VentureBeat | **Trending Score:** 3 Sources

<p>Attackers stole a long-lived npm access token belonging to the lead maintainer of <a href="https://www.npmjs.com/package/axios">axios</a>, the most popular HTTP client library in JavaScript, and used it to publish two poisoned versions that install a cross-platform remote access trojan. The malicious releases target macOS, Windows, and Linux. They were live on the npm registry for roughly three hours before removal.</p><p>Axios gets more than 100 million downloads per week. <a href="https://www.wiz.io/blog/axios-npm-compromised-in-supply-chain-attack">Wiz reports</a> it sits in approximately 80% of cloud and code environments, touching everything from React front-ends to CI/CD pipelines to serverless functions. <a href="https://www.huntress.com/blog/supply-chain-compromise-axios-npm-package">Huntress detected</a> the first infections 89 seconds after the malicious package went live and confirmed at least 135 compromised systems among its customers during the exposure window.</p><p>This is the third major <code>npm</code> supply chain compromise in seven months. Every one exploited maintainer credentials. This time, the target had adopted every defense the security community recommended.</p><h2><b>One credential, two branches, 39 minutes</b></h2><p>The attacker took over the <code>npm</code> account of @jasonsaayman, a lead axios maintainer, changed the account email to an anonymous ProtonMail address, and published the poisoned packages through <code>npm</code>’s command-line interface. That bypassed the project’s GitHub Actions CI/CD pipeline entirely.</p><p>The attacker never touched the Axios source code. Instead, both release branches received a single new dependency: <code>plain-crypto-js@4.2.1</code>. No part of the codebase imports it. The package exists solely to run a postinstall script that drops a cross-platform RAT onto the developer&#x27;s machine.</p><p>The staging was precise. Eighteen hours before the axios releases, the attacker published a clean version of <code>plain-crypto-js </code>under a separate <code>npm</code> account to build publishing history and dodge new-package scanner alerts. Then came the weaponized 4.2.1. Both release branches hit within 39 minutes. Three platform-specific payloads were pre-built. The malware erases itself after execution and swaps in a clean package.json to frustrate forensic inspection.</p><p><a href="https://www.stepsecurity.io/blog/axios-compromised-on-npm-malicious-versions-drop-remote-access-trojan">StepSecurity</a>, which identified the compromise alongside <a href="https://socket.dev/blog/axios-npm-package-compromised">Socket</a>, called it among the most operationally sophisticated supply chain attacks ever documented against a top-10 <code>npm</code> package.</p><h2><b>The defense that existed on paper</b></h2><p>Axios did the right things. Legitimate 1.x releases shipped through GitHub Actions using <code>npm</code>&#x27;s OIDC Trusted Publisher mechanism, which cryptographically ties every publish to a verified CI/CD workflow. The project carried SLSA provenance attestations. By every modern measure, the security stack looked solid.</p><p>None of it mattered. <a href="https://www.huntress.com/blog/supply-chain-compromise-axios-npm-package">Huntress dug into the publish workflow</a> and found the gap. The project still passed <code>NPM_TOKEN</code> as an environment variable right alongside the OIDC credentials. When both are present, <s>npm</s> defaults to the token. The long-lived classic token was the real authentication method for every publish, regardless of how OIDC was configured. The attacker never had to defeat OIDC. They walked around it. A legacy token sat there as a parallel auth path, and <code>npm</code>&#x27;s own hierarchy silently preferred it.</p><p>“From my experience at AWS, it’s very common for old auth mechanisms to linger,” said Merritt Baer, CSO at Enkrypt AI and former Deputy CISO at AWS, in an exclusive interview with VentureBeat. “Modern controls get deployed, but if legacy tokens or keys aren’t retired, the system quietly favors them. Just like we saw with SolarWinds, where legacy scripts bypassed newer monitoring.”</p><p>The maintainer <a href="https://github.com/axios/axios/issues/10604">posted on GitHub</a> after discovering the compromise: “I’m trying to get support to understand how this even happened. I have 2FA / MFA on practically everything I interact with.”</p><p><a href="https://www.endorlabs.com/learn/npm-axios-compromise">Endor Labs documented</a> the forensic difference. Legitimate<code> axios@1.14.0</code> showed OIDC provenance, a trusted publisher record, and a gitHead linking to a specific commit. Malicious <code>axios@1.14.1</code> had none. Any tool checking provenance would have flagged the gap instantly. But provenance verification is opt-in. No registry gate rejected the package.</p><h2><b>Three attacks, seven months, same root cause</b></h2><p>Three <code>npm</code> supply chain compromises in seven months. Every one started with a stolen maintainer credential.</p><p>The <a href="https://unit42.paloaltonetworks.com/npm-supply-chain-attack/">Shai-Hulud worm</a> hit in September 2025. A single phished maintainer account gave attackers a foothold that self-replicated across <a href="https://www.stepsecurity.io/blog/ctrl-tinycolor-and-40-npm-packages-compromised">more than 500 packages</a>, harvesting <code>npm</code> tokens, cloud credentials, and GitHub secrets as it spread. <a href="https://www.cisa.gov/news-events/alerts/2025/09/23/widespread-supply-chain-compromise-impacting-npm-ecosystem">CISA issued an advisory</a>. GitHub <a href="https://github.blog/security/supply-chain-security/our-plan-for-a-more-secure-npm-supply-chain/">overhauled <code>npm’</code>s entire authentication model</a> in response.</p><p>Then in January 2026, Koi Security’s <a href="https://www.koi.ai/blog/packagegate-6-zero-days-in-js-package-managers-but-npm-wont-act">PackageGate research</a> dropped six zero-day vulnerabilities across npm, <code>pnpm</code>, <code>vlt</code>, and Bun that punched through the very defenses the ecosystem adopted after Shai-Hulud. Lockfile integrity and script-blocking both failed under specific conditions. Three of the four package managers patched within weeks. npm closed the report.</p><p>Now axios. A stolen long-lived token published a RAT through both release branches despite OIDC, SLSA, and every post-Shai-Hulud hardening measure in place.</p><p><code>npm</code> shipped real reforms after Shai-Hulud. Creation of new classic tokens got deprecated, though pre-existing ones survived until a hard revocation deadline. FIDO 2FA became mandatory, granular access tokens were capped at seven days for publishing, and trusted publishing via OIDC gave projects a cryptographic alternative to stored credentials. Taken together, those changes hardened everything downstream of the maintainer account. What they didn’t change was the account itself. The credential remained the single point of failure.</p><p>“Credential compromise is the recurring theme across <code>npm</code> breaches,” Baer said. “This isn’t just a weak password problem. It’s structural. Without ephemeral credentials, enforced MFA, or isolated build and signing environments, maintainer access remains the weak link.”</p><h2><b>What npm shipped vs. what this attack walked past</b></h2><table><tbody><tr><td><p><b>What SOC leaders need</b></p></td><td><p><code><b>npm</b></code><b> defense shipped</b></p></td><td><p><b>vs. axios attack</b></p></td><td><p><b>The gap</b></p></td></tr><tr><td><p>Block stolen tokens from publishing</p></td><td><p>FIDO 2FA required. Lessons from log4j and event-stream show speed and clarity matter as much as the fix itself.”</p><ul><li><p><b>Check exposure. </b>Search lockfiles and CI logs for <code>axios@1.14.1</code>, <code>axios@0.30.4</code>, or <code>plain-crypto-js</code>. If found, preform a full rebuild.</p></li><li><p><b>Harden going forward. </b>Enforce <code>npm ci --ignore-scripts</code> in CI/CD. Require lockfile-only installs. Each different in execution, identical in root cause. <code>npm</code>’s security model still treats individual maintainer accounts as the ultimate trust anchor. Those accounts remain vulnerable to credential hijacking, no matter how many layers get added downstream.</p><p>“AI spots risky packages, audits legacy auth, and speeds SOC response,” Baer said. “But humans still control maintainer credentials. We mitigate risk. We don’t eliminate it.”</p><p>Mandatory provenance attestation, where manual CLI publishing is disabled entirely, would have caught this attack before it reached the registry. So would mandatory multi-party signing, where no single maintainer can push a release alone. Neither is enforced today. <code>npm</code> has signaled that disabling tokens by default when trusted publishing is enabled is on the roadmap. Until it ships, every project running OIDC alongside a legacy token has the same blind spot axios had.</p><p>The axios maintainer did what the community asked. A legacy token nobody realized was still active and undermined all of it.</p>
